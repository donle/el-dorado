# El Dorado 联机版 — 设计文档

> 《冲向黄金城》(The Quest for El Dorado / Wettlauf nach El Dorado, Reiner Knizia)
> 的可联机数字实现。TypeScript 全栈，LayaAir 2D 渲染，Node 权威服务器。

日期：2026-06-18
状态：设计已定，进入实现

---

## 1. 目标与范围

第一个版本（MVP）的成功标准：

- 2–4 名玩家通过**房间码**进入同一房间，可加入**简单 AI** 凑人数。
- 在一张**固定的六角格地图**上，用 deck-building 机制移动棋子赛跑。
- 完整实现核心规则循环：出牌移动 / 买牌 / 弃牌精简 / 抽牌 / 抵达终点判胜。
- 服务器是**权威裁判**，客户端只发操作意图、渲染服务器广播的状态。

非 MVP（后续迭代）：模块化拼接地图、洞穴扩展、全部 18 种市场牌的美术、断线重连完善、登录/战绩、观战。

## 2. 游戏规则（实现依据）

完整规则速览（详细版见研究报告，下面是实现必须严格遵守的要点）：

### 2.1 目标
2–4 人，deck-building 驱动棋子在六角格地图赛跑，第一个抵达黄金城（终点格）者获胜。

### 2.2 牌与符号
- **绿牌 = 砍刀(machete)** → 进绿色丛林格
- **蓝牌 = 船桨(paddle)** → 进蓝色水域格
- **黄牌 = 金币(coin)** → 进黄色村庄格，**也是买牌货币**
- **白牌 = 万能(joker)** → 出牌时选一种符号（不可在一次出牌里混用）
- **紫牌 = 行动牌** → 抽牌/移除牌/特殊移动等
- 牌属性：`{ symbol, power(数值), cost(价格), singleUse(是否用后移出) }`

### 2.3 起始牌库（每人 8 张，相同）
- 3 × 探险者 Explorer：绿/砍刀 1
- 4 × 旅行者 Traveller：黄/金币 1
- 1 × 水手 Sailor：蓝/船桨 1

手牌上限 **4**。

### 2.4 市场牌（18 种 × 3 张 = 54）

| 先锋 Pioneer | 砍刀 | 5 | 5 | |
| 巨型砍刀 Giant Machete | 砍刀 | 5 | 4* | 单次性 |
| 船长 Captain | 船桨 | 3 | 2 | |

### 2.5 地格类型
- 绿/蓝/黄：需对应符号单张牌、`power ≥ 格子cost`
- **灰色碎石**：出任意牌数量 = 符号数，入弃牌堆
- **红色大本营**：出任意牌数量 = 符号数，**永久移除**（精简牌库主要手段）
- **黑色山脉**：不可进入
- **路障**：首个通过者按需求出牌并**保留路障**（平局判定用）
- **起点格 / 终点格**

### 2.6 移动规则（最易错，必须精确）
1. 进入某格需要**单张牌**，符号匹配且 `power ≥ 该格 cost`。
2. **禁止多张牌凑同一格**。
3. 一张牌的余力可逐格穿过多个格子：每进一格扣该格 cost，剩余可继续；用不完作废，**不跨牌累积**。
4. 不能进入山脉，不能停在/穿过被占据的格子。

### 2.7 买牌
- 金币牌按面值，其他牌每张算 ½ 金币（两张非金币牌 = 1 金币）。
- 每回合最多买 **1** 张，买入的牌进**弃牌堆**。不找零。

### 2.8 回合结构
1. **出牌阶段**：任意张出牌用于移动 +（最多）买 1 张。
2. **弃牌阶段**：打出的牌进弃牌堆（被移除的除外）；手牌可自选保留或弃掉。
3. **抽牌阶段**：抽到手牌 4 张；牌库空则洗弃牌堆成新牌库。
- 单次性道具牌**用掉功能后移出游戏**（若只当 ½ 金币/碎石用则正常弃掉）。

### 2.9 胜负
- 抵达终点（3 个终点格之一）触发**最后一轮**，其余玩家各走最后一回合。
- 多人同轮抵达平局判定：路障数最多 → 持最高编号路障 → 先抵达者。

## 3. 架构

### 3.1 Monorepo（pnpm workspace）
```
el-dorando/
├── packages/
│   ├── core/      纯 TS 规则引擎（无渲染、无网络、确定性、单测覆盖）
│   ├── server/    Node + ws 权威服务器 + 房间管理 + AI bot
│   └── client/    LayaAir 2D 客户端
├── pnpm-workspace.yaml
└── package.json
```

**核心原则**：`core` 与渲染/网络完全解耦，前后端复用。服务器用它做权威裁判；客户端用它做合法性预校验与 UI 高亮。

### 3.2 core 规则引擎
- 纯函数 reducer：`applyAction(state, action): { state, events } | { error }`，确定性、无副作用。
- 数据模型：`Card` / `Hex`(轴向坐标 q,r) / `Player` / `Market` / `GameState`。
- Actions：`PlayCardsToMove` / `BuyCard` / `EndTurn` / `DiscardHand` 等。
- 卡牌数据集 `cards.ts`、地图数据 `maps/`（JSON）、随机数用可注入的种子化 RNG 保证确定性与可测。
- **TDD**：先写规则测试，再实现。

### 3.3 server 权威服务器
- Node + `ws`。`RoomManager` 持有每房间权威 `GameState`。
- 协议（WebSocket JSON）：
  - client→server：`CreateRoom` / `JoinRoom(code)` / `AddAI` / `StartGame` / `GameAction(intent)`
  - server→client：`RoomState`（房间内玩家列表）/ `GameSnapshot`（完整状态）/ `GameEvents`（动画事件）/ `Error`
- 状态同步：MVP **整快照广播**（简单可靠），事件列表辅助动画。
- 房间码：6 位短码，房主开局。
- AI bot：服务端启发式机器人，走同一 Action API（贪心朝终点 + 买最优牌）。
- 重连：MVP 凭 `playerId` 重新订阅快照。

### 3.4 client（Three.js 2.5D）
> 引擎决策更新（2026-06-18）：LayaAir 无可用的现代 npm 包（npm 上 `layaair` 是 2016 占位包，引擎仅经 IDE 分发），改用用户备选的 **Three.js**（npm/Vite 原生支持、活跃维护）。渲染层与 core/server 完全解耦，此切换不影响已完成部分。

- Three.js 正交相机俯视渲染六角棋盘（hex 多边形 mesh + 地形配色 + cost 数字），棋子为彩色圆柱；HTML 叠层做手牌/市场/房间 UI。
- 轴向坐标→像素的尖顶六角格渲染，地形配色。
- UI：手牌区、市场 6 槽、玩家面板（牌库/弃牌数、进度）、回合提示、房间码界面。
- 交互：点目标格→高亮可达路径与所需牌→选牌确认；点市场牌→选支付→买入；结束回合。
- 网络层：连 WebSocket，发意图、收快照渲染，事件驱动动画。

## 4. 开发顺序（增量、每步可验证）

1. **core 引擎**：卡牌/地图数据 + reducer + 完整单元测试（TDD，无 UI）。
2. **server**：房间 + WebSocket 协议 + 用 core 脚本驱动跑通一局。
3. **client**：LayaAir 渲染 + 交互，本地连服务器跑通真人对战。
4. **AI bot**：服务端 AI + 房间加 AI。
5. **打磨**：动画、断线重连、UI 细节。

---

## 5. 技术选型

- 语言：TypeScript 全栈；包管理 pnpm workspace
- 测试：Vitest（core 引擎为主）
- 网络：原生 `ws`
- 客户端构建：Vite + LayaAir npm 包
- 确定性：所有随机（洗牌）走可注入种子 RNG
